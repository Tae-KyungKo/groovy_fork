# 📊 Git Collaboration Guide

이 가이드는 프로젝트의 일관된 코드 관리와 원활한 협업을 위한 **브랜치 네이밍, 커밋 메시지 규칙 및 깃플로우(Git Flow) 전략**을 정의한 문서입니다. 팀원 모두가 동일한 규칙을 준수하여 생산성을 높일 수 있도록 적극 동참해 주시기 바랍니다.

---

## 1. 브랜치 & 커밋 메시지 타입 정의

작업의 목적을 한눈에 파악할 수 있도록 아래의 고유 타입을 사용합니다. 브랜치 생성 및 이슈 작성 시 가장 먼저 고려해야 할 기준입니다.

| 타입 (Type) | 설명 | 예시 |
| :--- | :--- | :--- |
| **`feat`** | 새로운 기능 추가 | 로그인 기능 구현, 회원가입 API 개발 |
| **`fix`** | 버그 수정 | 널 포인터 예외 처리, UI 깨짐 현상 수정 |
| **`docs`** | 문서 수정 | `README.md` 수정, API 명세서 업데이트 |
| **`refactoring`** | 코드 구조 개선 (기능 변경 없음) | 중복 코드 제거, 패키지 구조 변경 |
| **`test`** | 테스트 코드 추가 및 수정 | 단위 테스트 작성, Mock 데이터 추가 |
| **`chore`** | 기타 작업 (빌드 설정, 패키지 매니저 등) | `build.gradle` 의존성 추가, `.gitignore` 수정 |

## 2. 🚀 개발 프로세스 요약 (로그인 기능 예시)

새로운 기능을 개발하거나 수정할 때는 항상 다음의 3단계를 거칩니다.

### 1단계: 이슈(Issue) 생성
작업을 시작하기 전, 깃허브 Repository의 **Issues** 탭에서 작업 내용을 등록합니다.
* **형식:** `[타입]: 작업 내용`
* **예시:** `feat: 로그인 기능 개발`

### 2단계: 브랜치(Branch) 생성
항상 최신 상태의 기준 브랜치에서 분기하며, 소문자로 작성합니다.
* **형식:** `[타입]/[기능명]`
* **예시:** `feat/login`

### 3단계: 개발 완료 후 커밋(Commit)
코드 변경을 저장할 때는 반드시 영어로 명확히 작성하고, 앞서 생성한 이슈 번호를 링크합니다.
* **형식:** `[타입](#이슈번호): 커밋 메시지(영어)`
* **예시:** `feat(#12): implement login`

## 3. 🌊 깃플로우(Git Flow) 협업 전략

우리 프로젝트는 역할별로 브랜치를 분리하여 안정적인 배포 환경을 유지

### 📌 브랜치 역할 정의
* **`main` 브랜치 (배포 기준)**
    * 사용자가 실제로 사용하는 최종 안정 버전이 관리되는 브랜치입니다.
    * 일련의 기능 개발 및 버그 수정이 완벽히 끝나고 코드 리뷰를 통과한 코드만 병합됩니다.
* **`dev` 브랜치 (개발 기준)**
    * 다음 배포를 위한 개발이 통합되는 중심 브랜치입니다.
    * 모든 기능 브랜치의 출발점이자 종착지입니다.
* **기능 브랜치 (`feat/`, `fix/` 등)**
    * 당장의 작업(기능 개발, 버그 수정, 문서 작성 등)을 독립적으로 수행하는 브랜치입니다.

### 🔄 전체 개발 및 협업 흐름 (Work Flow)
[ 이슈 생성 ] ➔ [ dev에서 브랜치 생성 ] ➔ [ 기능 개발 & 커밋 ] 
                                                 ⬇
[ main 병합 ] ⬳ [ 코드 리뷰 & Merge ] ➔ [ dev로 PR 생성 ] ➔ [ Remote Push ]
